This operator was adversarially tested in Claude Code before submission. This file is the receipt: the actual ledger a live session produced for TRY_IT.md Test 3, and the rule defect the testing caught and fixed.

## The defect testing caught

First live run returned `INTEGRATION HALT` instead of the expected `MANDATORY LIFECYCLE PIVOT`. The operator wasn't wrong — the rules were. Rule 1 as originally written treated a *well-formed ID that simply isn't declared in the intent file* the same as a *malformed or conflicting ID*. That conflation meant every piece of drifting work halted at the integrity check before the drift scoring could ever run — making the PIVOT band dead code and contradicting the operator's own purpose (measuring drift, not just integrity).

**Fix:** Rule 1 now splits the two findings. Malformed/missing/conflicting IDs → integrity HALT, unchanged. Well-formed-but-undeclared IDs → routed to drift scoring (−0.20 CCI orphan deduction, +0.10 CDS undeclared dependency). The worked example, scoring rubric, and verification script were all updated to match, and the suite re-run: all cases and boundary probes pass.

The loop — spec → live test → defect surfaced → rule patched → retest → exact match — is the same governance discipline the operator itself enforces.
